# Feature 4 — Rare and Edge Cases

> **Poster wording.** "Occlusions, damaged goods, glare, spills, and
> people in the path. Generated on purpose so your model sees what
> real data rarely captures."

## What it is, in simple words

In production a robot fails on the **weird stuff** — a dented box, a
human walking through the camera's view, a spill on the floor, a glare
spot from the morning sun. Real-world data almost never has these
because:

1. They are *rare* by definition — you would need months of footage
   to see one of each.
2. Even when they happen, nobody labels them, so they cannot be used
   for training.

In simulation we **manufacture** these on purpose. We script a
"failure" — make this box dented, make this product half-occluded by
another product, drop a virtual human in front of the camera — and
dump the labelled frame just like we dump the easy ones.

A finished dataset looks like a normal detection or pose dataset but
each scene is **tagged** with which edge case it represents:

```
edge_cases_<project>/
├── images/
├── labels/
├── masks/
└── scenarios.json   ({frame_id: ["dented_box", "occlusion_50pct"], …})
```

## Who will use it

The customer's **perception engineer** and **safety / reliability
engineer**. The dataset goes in two places:

- **Training set** — to teach the model the rare cases exist.
- **Evaluation / regression set** — to *prove*, before shipping, that
  the model handles them.

Job titles: *Perception Engineer*, *ML Reliability Engineer*,
*Robotics Safety Engineer*, *QA / Validation Engineer*.

## What models the customer trains with it

- The same detectors / segmenters from
  [Feature 1](01-detection-images-and-masks.md), just trained on a
  harder mix.
- **Anomaly detectors** — small CNNs that flag "something is wrong
  here" without classifying it.
- **Out-of-distribution detectors** — `pytorch-ood` and similar; they
  use the edge-case set as a known OOD bucket during training.
- **Safety classifiers** — binary "is there a human in the frame
  right now?" gates that pause the robot.

## Libraries and frameworks involved

**On our side:**

- **Gazebo** or **Isaac Sim** with scripted scenario controllers.
- **Isaac Sim Replicator** — has built-in "scenario authoring" for
  occlusion / clutter / glare.
- **Blender + Python** — for "damaged goods" textures (dents,
  scratches, tears) applied as displacement maps.
- A `scenarios.yaml` config that describes each edge case we sweep.

**On the customer's side:**

- The same training stack from Feature 1 (Ultralytics, MMDetection,
  …). The edge-case dataset is just another folder of labelled
  frames.
- Optional: `albumentations` for data augmentation on top.

## What we ship (the formats)

- A normal labelled image dataset (COCO JSON or YOLO labels), **plus**
- A `scenarios.json` mapping each frame to the edge-case tag(s) it
  represents, **plus**
- A `regression_set/` subfolder — a curated, never-used-for-training
  set the customer runs on every model build to catch regressions.

## How we generate it (the methods)

- **Procedural scenes.** A config sweeps every combination — *5
  product types × 4 damage levels × 3 occlusion percentages × 6
  lighting setups × 2 humans-in-frame on/off* — and renders one
  scene per combination. That alone produces thousands of unique
  edge cases.
- **Domain randomisation.** Within each scenario we still vary
  lighting, textures, and camera pose so the model does not memorise
  "dented box always shows up at the same position."
- **Photo-real rendering.** Edge cases are usually *visual* — glare,
  shadow, dents. Isaac Sim's path tracer matters here.
- **Asset modification.** Damaged-goods variants are generated by
  applying displacement / normal maps to the clean CAD model. We
  build a small library of "damage textures" per product.

## Pain points this solves

- **"Our model demo'd great but our pilot site sees failures
  weekly."** Pilot sites have rare events the demo set never had.
  This feature builds the missing data.
- **"We can't reproduce a customer's failure in the lab."** We can
  re-create the scenario in sim, regenerate similar frames, and use
  them for the next training run.
- **"Safety review wants evidence the model handles humans in the
  cell."** This is the evidence — a labelled, held-out set with
  thousands of human-in-frame examples.

## What to say in a sales conversation

- "What failure modes are you seeing today in the field?" *We need
  the customer to name 3–5 specific edge cases. We rebuild each one
  in sim.*
- "Do you have any photos of past failures?" *Photos help us match
  the lighting / glare / occlusion patterns the customer's cell
  actually exhibits.*
- "Do you want this as training data, regression data, or both?"
  *Both is the typical answer.*
- "How will the customer's CI run the regression set?" *We can ship
  a small Python script that loads the model, runs the regression
  frames, prints mAP per scenario, and fails the build if any
  scenario regresses.*

## Typical scope and delivery

- **Inputs we need:** the list of edge cases the customer has hit
  (or expects), photos / videos of past failures when available, and
  the existing model so we can target its weak spots.
- **What we ship:** 2 000–10 000 edge-case frames split into
  `train/` and `regression/`, a `scenarios.json` index, and a
  one-page report telling the customer which edge cases their
  current model already passes vs fails.
- **Typical timeline:** 2–4 weeks. The CAD / damage-texture work is
  the bottleneck.
